# AnonWallet

A privacy-focused, multi-network testnet Ethereum wallet built with Next.js, TypeScript, and ethers.js. AnonWallet demonstrates two distinct privacy techniques — HD address hygiene and a self-implemented **ERC-5564 stealth address** system — with a clear, honest account of what each does and does not protect.

> **Educational / testnet only.** AnonWallet is a portfolio and learning project. It runs on Ethereum testnets, holds no real value, and is not audited. Do not use it with mainnet funds.

---

## Features

- **HD wallet** — BIP-39/BIP-44 derived addresses from a single seed phrase, encrypted at rest in the browser.
- **Multi-network** — Sepolia, Base Sepolia, Arbitrum Sepolia, and Polygon Amoy, with centralized RPC configuration and automatic failover across multiple endpoints.
- **Send / Receive / History** — real on-chain transaction broadcasting, QR receiving with a per-address switcher, and a filterable transaction history with live confirmation polling.
- **Stealth payments (ERC-5564)** — a from-scratch implementation of secp256k1 stealth addresses with view-tag fast scanning (details below).
- **Privacy Score** — a transparent breakdown of the wallet's privacy-relevant properties.
- **Local-first** — keys never leave the browser; all signing happens client-side.

## Privacy: an honest model

On-chain privacy is widely misunderstood, so AnonWallet is explicit about its threat model. The wallet combines two techniques that protect different things:

### 1. Fresh HD addresses — *address hygiene* (weak)

Deriving a new address per transaction makes casual linking slightly harder, but it is **not** strong privacy. The moment funds move between your own addresses — for example, consolidating balances — the addresses are publicly linked on-chain. Fresh addresses help only if funds are never combined and come from unrelated sources.

### 2. Stealth addresses (ERC-5564) — *recipient unlinkability* (real cryptographic privacy)

Stealth addresses break the link between a payment and the **recipient's identity** using elliptic-curve cryptography, not pooling. A recipient publishes a reusable **stealth meta-address**; senders derive a fresh one-time address from it for each payment. On-chain, these one-time addresses have no visible connection to the meta-address or to each other. Only the recipient, using their viewing key, can detect incoming payments, and only their spending key can move the funds.

**What stealth addresses protect:** that multiple incoming payments belong to the same person (recipient identity).

**What they do NOT protect:**
- the **sender** is fully visible (funding a stealth address is a normal, traceable transaction);
- the **amount** is visible;
- **spending** from a stealth address is itself a visible transaction;
- **consolidating** multiple stealth payments into one address re-links them.

### Stealth ≠ mixer

Stealth addresses are often confused with mixers (e.g. Tornado). They solve different problems:

| | Mixer | Stealth address |
|---|---|---|
| Hides | sender ↔ recipient link | recipient's identity |
| Mechanism | pooling + ZK membership proof | ECDH key derivation |
| Needs other users? | yes (anonymity set) | no (pure math) |
| Funds commingled? | yes | no |

Strong real-world privacy typically layers techniques: stealth addresses for receiving, a shielded pool for breaking sender links, and confidential amounts for hiding values.

## How the stealth system works

The implementation follows ERC-5564 scheme id 1 (secp256k1) and is written from scratch on `@noble/curves`, with no dependency on a stealth SDK.

**Keys.** Each user has a spending keypair and a viewing keypair. The public halves combine into a stealth meta-address: `st:eth:0x<spendingPubKey><viewingPubKey>` (two 33-byte compressed points).

**Sending.** The sender generates an ephemeral keypair and computes a shared secret via ECDH between their ephemeral private key and the recipient's viewing public key, hashed with keccak256 over the compressed point. The one-time stealth address is derived as `spendingPub + keccak256(sharedSecret)·G`. The sender publishes an announcement containing the ephemeral public key and a one-byte view tag.

**Receiving (scanning).** The recipient recomputes the shared secret from each announcement using their viewing private key. A one-byte **view tag** lets them skip roughly 255/256 of non-matching announcements with a single curve multiplication and hash, before doing the full check — the standard ERC-5564 scanning optimization.

**Spending.** For a detected payment, the recipient derives the controlling key as `(spendingPriv + keccak256(sharedSecret)) mod n`, which produces exactly the private key for the one-time stealth address, allowing the funds to be swept.

### Announcements: local vs. on-chain

The canonical ERC-5564 Announcer is a singleton deployed deterministically (CREATE2) at the same address on every chain, including Sepolia. For this educational build, announcements are stored **locally** (in the browser) so the full derive → fund → scan → sweep flow works on testnet without gas. The cryptography is identical to the on-chain version; only the announcement transport differs. Swapping to the on-chain Announcer is a self-contained upgrade that does not change the core crypto.

## Getting started

```bash
npm install
npm run dev
```

Open http://localhost:3000. Create a wallet, switch to a testnet, and fund an address from that network's faucet (paste the address only — never connect or import your seed into a faucet). The Sepolia PoW faucet (sepolia-faucet.pk910.de) works without a mainnet balance.

### Trying stealth payments

1. **My Address** — copy your stealth meta-address.
2. **Send** — paste a meta-address (your own, for testing), derive a one-time stealth address, then fund it. This both sends the funds and publishes the announcement.
3. **Scan** — detect the payment with your viewing key, then sweep it to any address.

Note: funding a stealth address requires publishing the announcement. Sending to a derived stealth address *without* announcing makes the payment undetectable by scanning — this is a core property of the protocol, not a bug.
