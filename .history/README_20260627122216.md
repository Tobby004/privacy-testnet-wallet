# AnonWallet

A privacy-focused, multi-network testnet Ethereum wallet built with Next.js, TypeScript, and ethers.js. AnonWallet implements **ERC-5564 stealth addresses** from scratch and pairs them with tools that make on-chain privacy *legible* — a behavioral Privacy Inspector and a Transaction Graph that visualizes how activity links (or unlinks) your addresses.

> **Educational / testnet only.** AnonWallet is a portfolio and learning project. It runs on Ethereum testnets, holds no real value, and is not audited. Do not use it with mainnet funds.

---

## Highlights

- **ERC-5564 stealth addresses** — a from-scratch secp256k1 implementation with view-tag fast scanning, on-chain announcements via the canonical Announcer, and signature-based key derivation.
- **Privacy Inspector** — audits your actual wallet activity and flags privacy leaks in plain English, with severity and concrete fixes.
- **Transaction Graph** — visualizes addresses and flows, with a toggle that shows how stealth links can re-expose you.
- **Multi-network** — Sepolia, Base Sepolia, Arbitrum Sepolia, Polygon Amoy, with centralized RPC config and automatic failover.
- **Real transactions** — client-side signing and broadcasting, ENS resolution, gas presets, live confirmation tracking.
- **Encrypted backup/restore** — export an encrypted file of your wallet and restore it anywhere.

## Privacy: an honest model

On-chain privacy is widely misunderstood, so AnonWallet is explicit about its threat model. It combines techniques that protect *different* things, and it never claims more than it delivers.

### Fresh HD addresses — *address hygiene* (weak)

Deriving a new address per transaction makes casual linking slightly harder, but it is **not** strong privacy. The moment funds move between your own addresses — for example, consolidating balances — the addresses are publicly linked on-chain.

### Stealth addresses (ERC-5564) — *recipient unlinkability* (real cryptographic privacy)

Stealth addresses break the link between a payment and the **recipient's identity** using elliptic-curve cryptography, not pooling. A recipient publishes a reusable stealth meta-address; senders derive a fresh one-time address from it for each payment. On-chain, these one-time addresses have no visible connection to the meta-address or to each other. Only the recipient, using their viewing key, can detect incoming payments, and only their spending key can move the funds.

**Protected:** that multiple incoming payments belong to the same person (recipient identity).

**NOT protected:**
- the **sender** is visible (funding a stealth address is a normal, traceable transaction);
- the **amount** is visible;
- **spending** from a stealth address is itself a visible transaction;
- **consolidating** multiple stealth payments into one address re-links them.

### Stealth ≠ mixer

| | Mixer (e.g. Tornado) | Stealth address |
|---|---|---|
| Hides | sender ↔ recipient link | recipient's identity |
| Mechanism | pooling + ZK membership proof | ECDH key derivation |
| Needs other users? | yes (anonymity set) | no (pure math) |
| Funds commingled? | yes | no |

Strong real-world privacy typically layers techniques: stealth addresses for receiving, a shielded pool for breaking sender links, and confidential amounts for hiding values.

## How the stealth system works

The implementation follows ERC-5564 scheme id 1 (secp256k1), written on `@noble/curves` with no stealth SDK dependency.

**Keys.** Each user has a spending keypair and a viewing keypair. The public halves combine into a meta-address: `st:eth:0x<spendingPubKey><viewingPubKey>` (two 33-byte compressed points). Keys are derived deterministically from a wallet signature, so they are recoverable from the seed phrase alone.

**Sending.** The sender generates an ephemeral keypair and computes a shared secret via ECDH between their ephemeral private key and the recipient's viewing public key, hashed with keccak256 over the compressed point. The one-time stealth address is `spendingPub + keccak256(sharedSecret)·G`. The sender publishes an announcement containing the ephemeral public key and a one-byte view tag.

**Receiving (scanning).** The recipient recomputes the shared secret from each announcement using their viewing private key. A one-byte view tag lets them skip roughly 255/256 of non-matching announcements with a single curve multiplication and hash before the full check — the standard ERC-5564 scanning optimization.

**Spending.** For a detected payment, the recipient derives the controlling key as `(spendingPriv + keccak256(sharedSecret)) mod n`, which is exactly the private key for the one-time stealth address.

### Announcements: local and on-chain

A toggle selects where announcements live:

- **Local** — stored in the browser. Free, instant, good for testing.
- **On-chain** — emitted to the canonical ERC-5564 Announcer (`0x55649E01B5Df198D18D95b5cc5051630cfD45564`, deployed deterministically on every chain including Sepolia). Detectable across devices; costs a second transaction.

The cryptography is identical in both modes; only the announcement transport differs.

## Privacy Inspector

A dedicated page that analyzes local wallet activity and reports privacy leaks, each with a severity, a plain-English explanation, and a concrete suggestion. Checks include:

- **Consolidation** — sending from multiple owned addresses to one destination links them (critical).
- **Stealth sweep re-linking** — sweeping multiple stealth payments to one address re-links them (critical).
- **Self-funded stealth** — funding your own stealth address from a known address (warning).
- **Sender reuse** — concentrating many sends on one address (warning).
- **Stealth usage** — a positive signal when stealth payments are in use (good).

The result is scored 0–100. The point is not the number but the feedback: the wallet tells you *how your own behavior leaks privacy* — the insight most wallets never surface.

---

## Transaction Graph

A network diagram of your activity: your addresses, stealth addresses, and external destinations as nodes; transactions as edges. A **"stealth links" toggle** is the signature interaction — turn it off to see the network as an outsider would (stealth addresses float free), turn it on to reveal the connections your own funding and sweeping create. It makes the Privacy Inspector's findings visual.

## Getting started

```bash
npm install
npm run dev
```

Open http://localhost:3000. Create a wallet, switch to a testnet, and fund an address from that network's faucet (paste the address only — never connect or import your seed into a faucet). The Sepolia PoW faucet (sepolia-faucet.pk910.de) works without a mainnet balance.

### Trying stealth payments

1. **My Address** — derive and copy your stealth meta-address.
2. **Send** — paste a meta-address (your own, for testing), derive a one-time stealth address, then fund it. Choose Local or On-chain announcements.
3. **Scan** — detect the payment with your viewing key, then sweep it.

Funding a stealth address requires publishing the announcement. Sending to a derived stealth address *without* announcing makes it undetectable by scanning — a core property of the protocol, not a bug. And funding your own stealth address from a known address links it to you; the Privacy Inspector will flag exactly this.

## Security notes

- Keys never leave the browser; all signing is client-side.
- The seed is encrypted at rest with PBKDF2 + XSalsa20-Poly1305; the wallet re-locks on refresh by design.
- Stealth keys are derived from a wallet signature, so they are recoverable from the seed phrase.
- Encrypted backups are protected by a separate password (PBKDF2 + XSalsa20-Poly1305) and are safe to store.
